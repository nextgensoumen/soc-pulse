# SOC Pulse - Project Goals & Workflow

## Project Objective
Build a unified Security Operations Center (SOC) project/application tools by combining 5 highly specialized security tools originally sourced from 5 different GitHub repositories. 

**CRITICAL NOTE ON MODULES:** These repositories were copied and renamed to our new module folder names. However, **their actual `README.md` files are the absolute source of truth** for how they function. All configurations must align with their original README documentation.

## Environment & Execution
- **Testing & Deployment Environment:** AWS Ubuntu Linux Machine
- **Master Orchestrator:** `soc-pulse-start.sh`
- **Setup Initialization:** All initialization scripts run from the `setup` folder

## STRICT DASHBOARD REQUIREMENT
The frontend React dashboard (in the `dashboard` folder) **MUST NOT** use mock or hardcoded data in the final iteration. All statuses, threat levels, and module activity shown in the dashboard **MUST BE entirely real**, parsed backwards from the actual logs, output, or data generated by the Ubuntu machine running these backend scripts.

## Backend Architecture Decision
After an architectural review, it has been permanently decided that the backend orchestrator will be built using **Node.js (Express + Socket.io/SSE)** rather than Python. 
- **Reasoning:** It natively merges with the React frontend codebase and directly integrates with the two Node.js-based scanner modules (`module-malware-scanner` and `module-webapp-scanner`).
- **Goal:** The Node.js backend must handle asynchronous `child_process` execution of the heavy Linux bash scripts without hanging, and must stream raw server terminal stdout/stderr logs in real-time to the React dashboard via WebSockets.

## Development Workflow Loop
This project relies on a continuous back-and-forth testing and debugging loop:
1. **Run:** Execute the master orchestrator (`soc-pulse-start.sh` or specific setup scripts) inside the AWS Ubuntu Machine.
2. **Report:** The user finds outputs and errors and reports them back to the AI (Antigravity).
3. **Fix:** The AI analyzes the error logs, debugging scripts, and writes the code fix.
4. **Update:** The AI updates the local codebase to implement the fix.
5. **Push:** When the user explicitly commands to **"upload"** or **"push to github"**, the AI will handle pushing the updated codebase to the target GitHub repo.
6. **Repeat:** Repeat this loop until the entire SOC Pulse ecosystem is perfectly stable.

> **PROJECT STATUS (COMPLETED):** As of the final commit, this back-and-forth debugging loop has entirely stabilized the orchestrator. All five modules are fully mapped and running flawlessly against live Ubuntu outputs without failing.

## Memory Tracking
The AI will continuously update files in this `memory` directory to accurately reflect the progress, solved bugs, current state, and pending tasks so that no context is ever lost.
